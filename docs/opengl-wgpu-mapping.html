Fixed-Function OpenGL → wgpu/WGSL

Wie die Dino-Buch-Beispiele (Kap. 1–2, LWJGL/GL11) dieselben Probleme lösen wie astronomore — nur mit dreißig Jahren Architektur dazwischen. Eine Stufe-für-Stufe-Gegenüberstellung entlang der Rendering-Pipeline.

Java / LWJGL — GL11 fixed-function & #version 130 GLSL
Modernes OpenGL (3.3+ Core / GLSL 330) — zum Aufklappen anklicken
Rust — wgpu 29 / WGSL (astronomore)

Jedes Codebeispiel trägt zusätzlich ein Etikett: Spec von der API-Version selbst festgelegt, Stil Wahl der Fenster-Bibliothek/des Buchs/des Projekts, austauschbar ohne Versionswechsel.

01 · fenster & render-loop

Wer ruft eigentlich einen Frame auf?

Beide haben eine blockierende Schleife irgendwo — die Frage ist nur, wer sie besitzt. Im Dino-Buch schreibt die App die while-Schleife selbst und ruft explizit alles auf: Zeitmessung, Zeichnen, Buffer-Swap. Bei winit gehört die Schleife der Bibliothek (event_loop.run_app()); unser Code sitzt nur noch als Callback window_event() darin und bekommt einen Frame zugeteilt, wenn winit einen für fällig hält.

Damit dieser Callback überhaupt wiederholt aufgerufen wird, muss jeder Frame selbst wieder einen weiteren anfordern — sonst würde die Schleife nach dem ersten Frame verstummen. Das ist die Aufgabe von request_redraw() am Anfang von render(): es plant keinen neuen Frame sofort, sondern reiht für die nächste Runde der winit-eigenen Schleife ein RedrawRequested-Event ein.

FlaecheSchwingt.javaStil · LWJGLrenderLoop() — Schleife im eigenen Code
// LWJGLBasisFenster.start():
initDisplay();
renderLoop();   // blockiert hier bis Close
Display.destroy();

// renderLoop() IST die Schleife — sie lebt
// komplett in unserem Code:
while (!Display.isCloseRequested()) {
    double t = (System.nanoTime()-start)/1e9;
    // ... Zustand setzen, zeichnen ...
    Display.update();  // swap + Events pollen,
                        // dann direkt weiter zur
                        // naechsten Iteration
}
GLFW · kein Vorlagen-BezugStil · GLFWgleiche Architektur wie links 
GLFWwindow* window = glfwCreateWindow(800, 600, "GL", 0, 0);
glfwMakeContextCurrent(window);
gladLoadGLLoader(...);  // Funktionszeiger laden —
                         // OpenGL selbst hat keine echte API

while (!glfwWindowShouldClose(window)) {
    float t = glfwGetTime();
    // ... Zustand setzen, zeichnen ...
    glfwSwapBuffers(window);   // swap
    glfwPollEvents();          // Events pollen
}
glfwTerminate();

Kein Fortschritt hier: Moderne Desktop-GL-Tutorials (GLFW/SDL) nutzen strukturell dieselbe App-eigene while-Schleife wie das Dino-Buch — nur der Fenster-Layer ist ausgetauscht. Der Sprung "App besitzt die Schleife" → "Bibliothek besitzt die Schleife" ist spezifisch der wgpu/winit-Schnitt, keine Konsequenz aus Shadern oder Core-Profile.

lib.rs:1519, 1446, 296Stil · winitrun_app() — Schleife in winit, wir nur Callback

// nur gehoert sie winit statt uns:
event_loop.run_app(&mut app)?;   // blockiert hier

// winit ruft bei jedem Durchlauf, wenn ein
// Redraw faellig ist, unseren Callback auf:
fn window_event(...) {
    match event {
        WindowEvent::RedrawRequested => {
            state.update();
            state.render();   // zeichnet DIESEN Frame
        }
        ...
    }
}

// am Anfang jedes render()-Aufrufs:
self.gpu.window.request_redraw();
// kein neuer Frame JETZT — nur eine Anmeldung
// fuer den NAECHSTEN Durchlauf der obigen
// run_app()-Schleife. Ohne diesen Aufruf würde
// nach diesem Frame kein RedrawRequested mehr
// kommen und die App stünde still.
Kernunterschied

Die Schleife verschwindet nicht — sie wandert nur aus unserem Code heraus in die Bibliothek. LWJGL und modernes GLFW-basiertes GL: wir schreiben while(...) und rufen glfwSwapBuffers()/Display.update() selbst am Ende jeder Iteration auf. winit: event_loop.run_app() ist die Schleife, und wir lösen die nächste Runde nicht durch eigenen Code-Fluss aus, sondern indem wir innerhalb des aktuellen Callback-Aufrufs request_redraw() rufen — eine Art "meld mich für die nächste Runde an", bevor winit weiterschaltet. Der Vorteil: dieselbe Schleife kann daneben auch Tastatur-/Maus-Events, Resize und (auf WASM) den Browser-eigenen requestAnimationFrame-Takt bedienen, ohne dass unser Code das koordinieren müsste.

02 · bildschirm löschen

POGL bündelt zwei State-Machine-Aufrufe zu einer Hilfsfunktion. In wgpu ist "löschen" kein Befehl mehr, den man mittendrin ausführt, sondern eine Eigenschaft, die man beim Eröffnen des Render-Passes deklariert.

glClear(GL_COLOR_BUFFER_BIT) ist ein Seiteneffekt, der irgendwo im Frame passiert und im POGL-Beispiel nur die Farbe trifft — kein Tiefenpuffer im Spiel. Die GPU weiß dadurch vorab, dass der alte Framebuffer-Inhalt irrelevant ist — auf Tile-basierten GPUs (Mobile, WebGL) spart das echte Bandbreite.

03 · geometrie zeichnen

Immediate Mode vs. Vertex/Index-Buffer

Das ist der größte architektonische Sprung im ganzen Vergleich. glBegin/glVertex/glEnd schickt bei jedem einzelnen Frame jeden Vertex einzeln über den Treiber zur GPU. wgpu lädt die Geometrie einmalig auf die GPU und referenziert sie danach nur noch.

  glVertex3f(-1, -1, -1); // Punkt A
  glVertex3f( 1, -1, -1); // Punkt B
  glVertex3f( 1,  1, -1); // Punkt C
  glVertex3f(-1,  1, -1); // Punkt D
  // ... 5 weitere Seiten, jeden Frame neu
glEnd();

Der Immediate-Mode → Buffer-Sprung ist der eigentliche "moderne" Schnitt in der GPU-Programmierung insgesamt, nicht etwas, das erst mit wgpu kommt. Nur die Typsicherheit (Layout wird zur Compile-Zeit geprüft statt per glVertexAttribPointer-Offset von Hand) ist wgpu-spezifisch.

Kernunterschied

Immediate Mode kennt keinen Unterschied zwischen "Geometrie beschreiben" und "Geometrie zeichnen" — beides passiert gleichzeitig, jeden Frame, als ein Strom einzelner Vertex-Aufrufe. wgpu trennt das strikt: Vertex-/Index-Buffer sind persistenter GPU-Speicher (einmal in Mesh::sphere hochgeladen), das Zeichnen pro Frame ist nur noch das Binden dieser Referenzen plus dreier Bind Groups (Textur, Kamera, Modell — siehe Stufe 5) und ein Draw-Call. Deshalb tessellieren wir die Kugel auch nur bei Parameteränderung neu (rebuild_mesh), nicht pro Frame.

04 · transformationskette

M = I · P · T · R · S — Stack vs. Ausdruck

Folie 41/42 zeigt die Kette M=I; M*P; M*T; M*R; M*S als flache Folge mutierender Aufrufe auf einem globalen Matrix-Stack. Um sie mit der rechten Seite vergleichbar zu machen, wird sie hier nicht in der Reihenfolge der Folie abgetippt, sondern nach denselben drei Rollen umsortiert, die astronomore explizit benennt: Projektion, View, Modell.

FlaecheSchwingt.java:46–58, umsortiertSpec · Compat-Profileein Stack, drei Rollen
glLoadIdentity();                       // M = I

// Projektion (Kamera- -> Clip-Space):

// Beispiel ganz — die Kamera steht fest im Ursprung.
// Das ist eine Eigenschaft dieses Buch-Beispiels, nicht
// von GL11 selbst: eine bewegte Kamera waere genauso
// im selben Stack, nur zusaetzlich vor T/R/S — und
// genauso unbenannt/unsichtbar.

// Modell (Objekt- -> Welt-Space):
glTranslated(0, 0, -4);                 // M = M*T
glRotatef(sin(t)*120f, 0,1,0);          // M = M*R
glScaled(0.8, 0.8, 0.8);                // M = M*S

POGL.renderViereck();  // v' = M*v = P*(T*R*S)*v, pro Vertex

Die Sprache wechselt, das Konzept "explizite CPU-Matrix statt globalem Stack" nicht.

Kernunterschied

Die Reihenfolge-Logik ("rechts wirkt zuerst") ist identisch — das ist dieselbe Mathematik. Der eigentliche Unterschied zeigt sich erst, wenn man beide Seiten in dieselben drei Rollen sortiert: astronomore hat für View und Modell zwei getrennte, benannte Ausdrücke (proj * view einmal pro Frame in CameraUniform::update_view_proj, world * spin * scale einmal pro Körper in ModelUniform::new) — GL11 dagegen kennt diese Trennung als Konzept gar nicht: Der Fixed-Function-Stack heißt nicht zufällig GL_MODELVIEW (im Gegensatz zu GL_PROJECTION) — View und Modell sind von Anfang an ein Stack, ein Name, eine Matrix. Im Beispiel oben fällt die View sogar ganz weg (Kamera fest im Ursprung), was im Code unsichtbar bleibt; man kann ihm nicht ansehen, ob glTranslated(0,0,-4) "die Kamera wegbewegen" oder "das Objekt platzieren" bedeutet — beides läuft im selben Stack. glam erzwingt diese Unterscheidung von Anfang an, weil view und model zwei separate Variablen mit eigenem Typ sind. Modernes GL (Mitte) trifft dieselbe Entscheidung wie astronomore — der Stack ist also ein Fixed-Function-Merkmal, kein "OpenGL vs. wgpu"-Merkmal. Dass dafür GLM bzw. glam zum Einsatz kommen, ist wiederum keine API-Vorgabe: beides sind austauschbare Drittanbieter-Bibliotheken (man könnte genauso eine andere Mathe-Bibliothek oder eigene float[16]-Multiplikation schreiben) — nur eben eine, die in beiden Ökosystemen zum Quasi-Standard wurde.

05 · daten in den shader bringen

Built-ins vs. explizite Bind Groups

Die Torus-Shader nutzen GLSL-Built-ins, die im Fixed-Function-Kontext "einfach da sind" — gl_ModelViewProjectionMatrix, gl_Normal. Sie sind nur deshalb befüllt, weil der CPU-Code aus Stufe 4 genau diesen Matrix-Stack (glLoadIdentity/glFrustum/...) bespielt — die Built-ins sind also eine direkte Konsequenz der Stack-Wahl, kein separates Feature. WGSL kennt so etwas nicht: jede Matrix, jeder Wert muss über eine struct deklariert und einer Bind Group zugeordnet werden.

torusambientdiffusespecularfinal.vsSpec · Compat-GLSLimplizite Built-ins
#version 130
uniform float ambient, diffuseWeight, specularWeight;
uniform vec3 lightDirection;

void main() {
  vec3 n = gl_NormalMatrix * gl_Normal;      // woher?
  gl_Position = gl_ModelViewProjectionMatrix
                * gl_Vertex;                 // woher?
}
GLSL 330 core · kein Vorlagen-BezugSpec · GLSL 330 corefast wie rechts 
#version 330 core
layout(location=0) in vec3 aPos;
layout(location=1) in vec2 aTexCoord;
layout(location=2) in vec3 aNormal;

uniform mat4 model, view, projection;

out vec3 Normal;
out vec3 FragPos;

void main() {
  FragPos = vec3(model * vec4(aPos, 1.0));
  Normal  = mat3(transpose(inverse(model))) * aNormal;
  gl_Position = projection * view * vec4(FragPos, 1.0);
}

Hier konvergiert es fast: gl_ModelViewProjectionMatrix ist im Core-Profile komplett verschwunden — Position, Uniforms und Normale müssen genauso explizit deklariert werden wie in WGSL. Der Unterschied zu rechts ist nur die Bindung: glGetUniformLocation(program, "model") ist ein loser String-Lookup ohne Layout-Prüfung zur Compile-Zeit — im Gegensatz zu WGSLs nummerierten @group/@binding, die naga vorab validiert.

Nichts kommt "von irgendwoher": jede Bind Group in astronomore hat eine feste, dokumentierte Nummer (0 Textur, 1 Kamera, 2 Modell, 3 Szene-Eigenschaften — siehe Kommentar oben in shader.wgsl). Dass es Bind Groups geben muss, ist WGSL-Spec; welche Nummer wofür steht, ist reine astronomore-Konvention — WGSL selbst schreibt keine Bedeutung für Gruppe 0/1/2/3 vor, genau wie GLSL keine Bedeutung für den Uniform-Namen "lightDirection" vorschreibt. Der Preis für die Bind-Group-Pflicht ist mehr Boilerplate; der Gewinn ist, dass der Shader unabhängig vom Rest der Pipeline lesbar und typgeprüft ist (naga-Validierung statt Laufzeit-Crash). Modernes GLSL (Mitte) löst das "woher?"-Problem schon — nur die Absicherung dagegen bleibt schwächer als bei WGSL.

06 · beleuchtung

Per-Vertex-Phong vs. Per-Fragment-Blinn-Phong

Beide Beispiele rechnen ambient + diffus + spekular. Der Torus-Shader tut das pro Vertex und interpoliert die fertige Farbe über das Dreieck (Gouraud-Shading) — sichtbar an den Facetten bei grober Tessellierung. astronomore rechnet pro Fragment und nutzt den Blinn-Phong-Halfway-Vektor statt reflect().

torusambientdiffusespecularfinal.vsStil · Buch-BeispielPhong, pro Vertex
vec3 l = normalize(lightDirection - (gl_ModelViewProjectionMatrix
                                       * gl_Vertex).xyz);
vec3 n = gl_NormalMatrix * gl_Normal;
vec3 v = -normalize((gl_ModelViewMatrix * gl_Vertex).xyz);
vec3 r = reflect(l, n);
float specular = max(0., pow(dot(r,v), specularPow));
float diffuse  = dot(n, l);
color = vec4(1) * (ambient + diffuse*diffuseWeight
                          + specular*specularWeight);
// -> als "flat" color interpoliert, .fs gibt sie nur aus
GLSL 330 core · kein Vorlagen-BezugStil · Konventionfast wie rechts 
// im Fragment-Shader (Standard seit LearnOpenGL & co.):
vec3 N = normalize(Normal);
vec3 L = normalize(lightPos - FragPos);
vec3 V = normalize(viewPos - FragPos);
vec3 H = normalize(L + V);           // Halfway-Vektor

float diff = max(dot(N, L), 0.0);
float spec = pow(max(dot(N, H), 0.0), shininess);
FragColor = vec4(ambient + diff*diffuseColor
                         + spec*specColor, 1.0);

Hier konvergiert es, aber als Konvention, nicht als Zwang: Nichts in GLSL 330 core verbietet Vertex-Lighting — man könnte den linken Torus-Shader unverändert unter #version 330 core kompilieren (bis auf die entfernten Built-ins). Dass moderne Tutorials trotzdem fast immer Fragment-Shading mit Halfway-Vektor zeigen, ist gelernte Praxis der Community, keine Sprachregel. Spec-seitig ist nur gl_NormalMatrix wirklich weg — die Normalenmatrix muss jetzt explizit gebaut werden, egal für welchen Shading-Stil man sich entscheidet.

shader.wgsl:94–110Stil · astronomoreBlinn-Phong, pro Fragment
let N = normalize(in.body_normal.xyz);
let L = normalize(light_pos - in.body_position.xyz);
let V = normalize(camera.camera_pos - in.body_position.xyz);
let H = normalize(L + V);           // Halfway-Vektor

let diffuse  = light_color * scene_props.diffuse_factor
                * max(dot(L, N), 0.0);
let specular = light_color * scene_props.specular_intensity
                * pow(max(dot(H, N), 0.0), scene_props.shininess);
Kernunterschied

Gouraud (Vertex) vs. Phong-Shading (Fragment) ist ein klassischer Buch-Gegensatz (Kap. 6), aber in keiner der drei Spalten eine API-Vorschrift — alle drei Beleuchtungs-Panels sind hier als "Stil" markiert, nicht als "Spec". Bei 128×64 Segmenten pro Kugel wäre Vertex-Shading hier vermutlich kaum vom Fragment-Ergebnis zu unterscheiden — aber die Blinn-Halfway-Formel pow(dot(H,N), shininess) ist zusätzlich numerisch stabiler als reflect() bei streifenden Winkeln, was bei der Sonne als Punktlicht direkt sichtbar wird.

07 · material: farbe oder textur

08 · shader laden

.vs/.fs-Dateipaare vs. eine validierte .wgsl-Datei

Kapitel 2 trennt Vertex- und Fragment-Shader in eigene Dateien (torus*.vs / torus*.fs). ShaderUtilities.java liest davon nur den Text ein und prüft nach dem Linken den Erfolg über einen GL-Info-Log — das eigentliche glCompileShader/glLinkProgram passiert in den jeweiligen Beispiel-Klassen selbst (hier nicht gezeigt). Dass es überhaupt zwei getrennte Helfer-Funktionen sind (readShadercode, testShaderProgram), ist Buch-eigene Code-Organisation — die GL-Aufrufe selbst sind seit GL 2.0 unverändert. astronomore validiert dagegen vor der GPU, mit exakter Fehlerposition im Quelltext.

Kernunterschied

In allen drei Welten prüft eine Hilfsfunktion den Erfolg — aber zu unterschiedlichen Zeitpunkten. GL11 und modernes GL (links, Mitte) fragen die GPU nach Compile und/oder Link, ob es geklappt hat (Fehler landet im GL-Info-Log als String) — das ist in beiden Fällen dieselbe, seit GL 2.0 unveränderte Spec-API. Rechts ist die Sache subtiler, als sie aussieht: wgpu nutzt naga ohnehin intern für jeden Shader, mit oder ohne validate_wgsl — die Bibliothek würde einen kaputten Shader also auch ohne diesen Aufruf ablehnen. Der eigentliche Mehrwert von validate_wgsl ist astronomores eigene Entscheidung, naga zusätzlich, separat und vorab aufzurufen, um an einen sauberen, miette-formatierten Fehler mit Source-Span zu kommen statt an wgpus eigene (deutlich kargere) Validierungsmeldung.
