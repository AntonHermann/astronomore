01 · fenster & render-loop

Wer ruft eigentlich einen Frame auf?

Beide haben eine blockierende Schleife irgendwo — die Frage ist nur, wer sie besitzt. Im Dino-Buch schreibt die App die while-Schleife selbst und ruft explizit alles auf: Zeitmessung, Zeichnen, Buffer-Swap. Bei winit gehört die Schleife der Bibliothek (event_loop.run_app()); unser Code sitzt nur noch als Callback window_event() darin und bekommt einen Frame zugeteilt, wenn winit einen für fällig hält.

Damit dieser Callback überhaupt wiederholt aufgerufen wird, muss jeder Frame selbst wieder einen weiteren anfordern — sonst würde die Schleife nach dem ersten Frame verstummen. Das ist die Aufgabe von request_redraw() am Anfang von render(): es plant keinen neuen Frame sofort, sondern reiht für die nächste Runde der winit-eigenen Schleife ein RedrawRequested-Event ein.

FlaecheSchwingt.javarenderLoop() — Schleife im eigenen Code
// LWJGLBasisFenster.start():
initDisplay();
renderLoop();   // blockiert hier bis Close
Display.destroy();

// renderLoop() IST die Schleife — sie lebt
// komplett in unserem Code:
while (!Display.isCloseRequested()) {
    double t = (System.nanoTime()-start)/1e9;
    // ... Zustand setzen, zeichnen ...
    Display.update();  // swap + Events pollen,
                        // dann direkt weiter zur
                        // naechsten Iteration
}
lib.rs:1519, 1446, 296run_app() — Schleife in winit, wir nur Callback
// run() — DIES ist die eigentliche Schleife,
// nur gehoert sie winit statt uns:
event_loop.run_app(&mut app)?;   // blockiert hier

// winit ruft bei jedem Durchlauf, wenn ein
// Redraw faellig ist, unseren Callback auf:
fn window_event(...) {
    match event {
        WindowEvent::RedrawRequested => {
            state.update();
            state.render();   // zeichnet DIESEN Frame
        }
        ...
    }
}

// am Anfang jedes render()-Aufrufs:
self.gpu.window.request_redraw();
// kein neuer Frame JETZT — nur eine Anmeldung
// fuer den NAECHSTEN Durchlauf der obigen
// run_app()-Schleife. Ohne diesen Aufruf würde
// nach diesem Frame kein RedrawRequested mehr
// kommen und die App stünde still.
Kernunterschied

Die Schleife verschwindet nicht — sie wandert nur aus unserem Code heraus in die Bibliothek. LWJGL: wir schreiben while(...) und rufen Display.update() selbst am Ende jeder Iteration auf. winit: event_loop.run_app() ist die Schleife, und wir lösen die nächste Runde nicht durch eigenen Code-Fluss aus, sondern indem wir innerhalb des aktuellen Callback-Aufrufs request_redraw() rufen — eine Art "meld mich für die nächste Runde an", bevor winit weiterschaltet. Der Vorteil: dieselbe Schleife kann daneben auch Tastatur-/Maus-Events, Resize und (auf WASM) den Browser-eigenen requestAnimationFrame-Takt bedienen, ohne dass unser Code das koordinieren müsste.

02 · bildschirm löschen

Clear-Color: Befehl vs. Deklaration

POGL bündelt zwei State-Machine-Aufrufe zu einer Hilfsfunktion. In wgpu ist "löschen" kein Befehl mehr, den man mittendrin ausführt, sondern eine Eigenschaft, die man beim Eröffnen des Render-Passes deklariert.

POGL.javaclearBackgroundWithColor
public static void clearBackgroundWithColor(
    float r, float g, float b, float a) {
  glClearColor(r, g, b, a);
  glClear(GL_COLOR_BUFFER_BIT);
}

    load: LoadOp::Clear(Color {
      r: self.view.background_color[0],

  },

Kernunterschied

glClear ist ein Seiteneffekt, der irgendwo im Frame passiert. LoadOp::Clear ist Teil der Beschreibung des Render-Passes selbst — die GPU weiß dadurch vorab, dass der alte Framebuffer-Inhalt irrelevant ist, was auf Tile-basierten GPUs (Mobile, WebGL) echte Bandbreite spart.

03 · geometrie zeichnen

Immediate Mode vs. Vertex/Index-Buffer

Das ist der größte architektonische Sprung im ganzen Vergleich. glBegin/glVertex/glEnd schickt bei jedem einzelnen Frame jeden Vertex einzeln über den Treiber zur GPU. wgpu lädt die Geometrie einmalig auf die GPU und referenziert sie danach nur noch.

POGL.javaWürfel, immediate mode
glBegin(GL_QUADS);
  glVertex3f(-1, -1, -1); // Punkt A
  glVertex3f( 1, -1, -1); // Punkt B
  glVertex3f( 1,  1, -1); // Punkt C
  glVertex3f(-1,  1, -1); // Punkt D
  // ... 5 weitere Seiten, jeden Frame neu
glEnd();
mesh.rs · celestial_body.rs:210einmalig hochladen, pro Frame binden
// einmalig, beim Erzeugen des Körpers:
let mesh = Mesh::sphere(device, 128, 64);
// -> vertex_buffer, index_buffer auf der GPU

// pro Frame, nur noch eine Referenz:
pass.set_vertex_buffer(0, mesh.vertex_buffer.slice(..));
pass.set_index_buffer(mesh.index_buffer.slice(..), ...);
pass.draw_indexed(0..mesh.num_elements, 0, 0..1);
Kernunterschied

Immediate Mode kennt keinen Unterschied zwischen "Geometrie beschreiben" und "Geometrie zeichnen" — beides passiert gleichzeitig, jeden Frame. wgpu trennt das strikt: Vertex-/Index-Buffer sind persistenter GPU-Speicher, das Zeichnen ist nur noch ein Draw-Call, der auf bereits vorhandene Daten zeigt. Deshalb tessellieren wir die Kugel auch nur bei Parameteränderung neu (rebuild_mesh), nicht pro Frame.

04 · transformationskette

M = I · P · T · R · S — Stack vs. Ausdruck

Folie 41/42 zeigt die Kette M=I; M*P; M*T; M*R; M*S als Folge mutierender Aufrufe auf einem globalen Matrix-Stack, rückwärts gelesen. astronomore rechnet dieselbe Mathematik — aber als benannten, expliziten Ausdruck auf der CPU, der dann als fertige Matrix zur GPU hochgeladen wird.

FlaecheSchwingt.java:46–58impliziter Matrix-Stack
glLoadIdentity();                    // M=I
glFrustum(-1,1, -1,1, 3,10);         // M*P
glTranslated(0, 0, -4);              // M*T
glRotatef(sin(t)*120f, 0,1,0);       // M*R
glScaled(0.8, 0.8, 0.8);             // M*S
POGL.renderViereck();                // v' = M*v, pro Vertex
celestial_body.rs:24–30 · camera.rs:135,164explizite glam-Matrix
// Projektion & View (camera.rs):
Mat4::perspective_rh(fov_y, aspect, near, far);
Mat4::look_at_rh(camera_pos, target, Vec3::Y);

// Modell-Transform (celestial_body.rs):
let scale = Mat4::from_scale(Vec3::splat(radius));
let model = world_transform * spin_transform * scale;
// -> einmal pro Frame in Uniform-Buffer geschrieben
Kernunterschied

Die Reihenfolge-Logik ("rechts wirkt zuerst") ist identisch — das ist dieselbe Mathematik. Der Unterschied ist, wo sie lebt: GL11 mutiert einen unsichtbaren globalen Zustand mit Seiteneffekt-Aufrufen; glam multipliziert benannte Mat4-Werte zu einem Ausdruck, der nichts Globales anfasst und sich debuggen/testen lässt, bevor er die GPU überhaupt erreicht.

05 · daten in den shader bringen

Built-ins vs. explizite Bind Groups

Die Torus-Shader nutzen GLSL-Built-ins, die im Fixed-Function-Kontext "einfach da sind" — gl_ModelViewProjectionMatrix, gl_Normal. WGSL kennt so etwas nicht: jede Matrix, jeder Wert muss über eine struct deklariert und einer Bind Group zugeordnet werden.

torusambientdiffusespecularfinal.vsimplizite Built-ins
#version 130
uniform float ambient, diffuseWeight, specularWeight;
uniform vec3 lightDirection;

void main() {
  vec3 n = gl_NormalMatrix * gl_Normal;      // woher?
  gl_Position = gl_ModelViewProjectionMatrix
                * gl_Vertex;                 // woher?
}
shader.wgsl:1–42explizite Bind Groups
// group 1: Kamera
struct CameraUniform { view_proj: mat4x4<f32>, camera_pos: vec3<f32> }
@group(1) @binding(0) var<uniform> camera: CameraUniform;

// group 2: Modell-Transform
struct ModelUniform { model_to_world_transform: mat4x4<f32> }
@group(2) @binding(0) var<uniform> model_transform: ModelUniform;
Kernunterschied

Nichts kommt "von irgendwoher": jede Bind Group in astronomore hat eine feste, dokumentierte Nummer (0 Textur, 1 Kamera, 2 Modell, 3 Szene-Eigenschaften — siehe Kommentar oben in shader.wgsl). Der Preis dafür ist mehr Boilerplate; der Gewinn ist, dass der Shader unabhängig vom Rest der Pipeline lesbar und typgeprüft ist (naga-Validierung statt Laufzeit-Crash).

06 · beleuchtung

Per-Vertex-Phong vs. Per-Fragment-Blinn-Phong

Beide Beispiele rechnen ambient + diffus + spekular. Der Torus-Shader tut das pro Vertex und interpoliert die fertige Farbe über das Dreieck (Gouraud-Shading) — sichtbar an den Facetten bei grober Tessellierung. astronomore rechnet pro Fragment und nutzt den Blinn-Phong-Halfway-Vektor statt reflect().

torusambientdiffusespecularfinal.vsPhong, pro Vertex
vec3 l = normalize(lightDirection - (gl_ModelViewProjectionMatrix
                                       * gl_Vertex).xyz);
vec3 n = gl_NormalMatrix * gl_Normal;
vec3 v = -normalize((gl_ModelViewMatrix * gl_Vertex).xyz);
vec3 r = reflect(l, n);
float specular = max(0., pow(dot(r,v), specularPow));
float diffuse  = dot(n, l);
color = vec4(1) * (ambient + diffuse*diffuseWeight
                          + specular*specularWeight);
// -> als "flat" color interpoliert, .fs gibt sie nur aus
shader.wgsl:94–110Blinn-Phong, pro Fragment
let N = normalize(in.body_normal.xyz);
let L = normalize(light_pos - in.body_position.xyz);
let V = normalize(camera.camera_pos - in.body_position.xyz);
let H = normalize(L + V);           // Halfway-Vektor

let diffuse  = light_color * scene_props.diffuse_factor
                * max(dot(L, N), 0.0);
let specular = light_color * scene_props.specular_intensity
                * pow(max(dot(H, N), 0.0), scene_props.shininess);
Kernunterschied

Gouraud (Vertex) vs. Phong-Shading (Fragment) ist ein klassischer Buch-Gegensatz (Kap. 6). Bei 128×64 Segmenten pro Kugel wäre Vertex-Shading hier vermutlich kaum vom Fragment-Ergebnis zu unterscheiden — aber die Blinn-Halfway-Formel pow(dot(H,N), shininess) ist zusätzlich numerisch stabiler als reflect() bei streifenden Winkeln, was bei der Sonne als Punktlicht direkt sichtbar wird.

07 · material: farbe oder textur

glColor3d als globaler State vs. Uniform-Flag

Immediate Mode setzt die aktuelle Zeichenfarbe als globalen State, der für alle folgenden glVertex-Aufrufe gilt, bis er erneut geändert wird. astronomore trifft dieselbe Entscheidung (Textur oder Flachfarbe) pro Fragment über ein Uniform-Flag.

HintergrundUmschalten.javaglobaler Farb-State
glColor3d(0.05, 0.39, 0.51);   // gilt ab jetzt für alle
glLineWidth(3.0f);             // folgenden glVertex-Aufrufe,
POGL.renderViereck();          // bis erneut glColor3d() kommt
shader.wgsl:88–89Uniform-Flag pro Fragment
let tex_color = textureSample(t_diffuse, s_diffuse, in.tex_coords).rgb;
let base_color = select(
  scene_props.object_color.xyz,   // wenn use_texture == 0
  tex_color,                      // wenn use_texture != 0
  scene_props.use_texture != 0u,
);
Kernunterschied

textureSample wird hier bewusst unbedingt aufgerufen, auch wenn das Ergebnis danach verworfen wird (Kommentar im Original-Shader) — ein WebGL/WGSL-spezifisches Detail: der Textur-Sample-Aufruf muss in "uniform control flow" liegen, darf also nicht in einem if-Block stehen, selbst wenn die Bedingung zur Laufzeit für alle Fragmente gleich ausfällt. Ein Fixed-Function-Äquivalent dazu gibt es schlicht nicht, weil glColor3d nie mit Texture-Sampling konkurriert.

Kapitel 2 trennt Vertex- und Fragment-Shader in eigene Dateien (torus*.vs / torus*.fs), geladen und kompiliert über ShaderUtilities.java. astronomore hält beide Stufen in einer .wgsl-Datei und validiert sie vor dem Erzeugen des Shader-Moduls.

Ein Shader-Fehler in GL11 zeigt sich oft erst als schwarzes Dreieck oder ein Log-String zur Laufzeit. naga parst WGSL vorab und liefert einen source-span-genauen Fehler bevor die GPU ihn je sieht — der Unterschied zwischen "Debuggen am Treiber-Log" und "Compiler-Fehler mit Zeilennummer".
