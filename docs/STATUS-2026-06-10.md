# Statusbericht & Retrospektive — 10. Juni 2026

## 1. Wo das Projekt steht

Phasen 1–3 sind abgeschlossen, Phase 5 (VSOP87-Ephemeriden, Zeitsteuerung) zu
großen Teilen ebenfalls. Das Ergebnis nach effektiv ~10 intensiven Tagen
(20.–31. Mai): ein vollständiges 10-Körper-Sonnensystem mit Blinn-Phong-
Beleuchtung, NASA-Texturen, zwei Kameramodi, Touch-Navigation, egui-Overlay,
Shader-Hot-Reload, Benchmark-System — nativ **und** im Browser, deployed via
GitHub Pages. ~4 800 Zeilen Rust/WGSL, 50 Commits, 19 PRs.

Das ist für den Zeitraum ein bemerkenswerter Umfang. Als Portfolio-Stück ist
das Projekt schon jetzt vorzeigbar; die fehlenden Punkte (Sterne, Saturnringe,
Bloom) sind Politur, kein Fundament.

Seit dem 31. Mai ruht das Projekt (~10 Tage).

## 2. Was gut gelaufen ist

- **Die Arbeitsteilung hat im Kern funktioniert.** Die Commit-Historie zeigt:
  Blinn-Phong, Specular-Draft, Transform-Klarheit, Tessellierungs-Regler kamen
  von Anton; WASM/Android-Debugging (PRs #5–#9, #16–#19), Touch-UI, Benchmark-
  System, Lokalisierung kamen aus Claude-Sessions. Genau die gewünschte
  Trennung „Grafik lernen vs. Drumherum delegieren".
- **Architektur ist sauber und erweiterbar:** `OrbitalModel`-Enum, deklaratives
  `planets.rs`, `BodyId`-Newtype, Pipelines ausgelagert. Neue Features docken
  bisher ohne Umbauten an.
- **Guardrails existieren:** `just san`, naga-Shader-Validierung mit miette,
  Pre-Commit-Benchmarks.

## 3. Wo es hakt — Befunde aus der Repo-Archäologie

Die folgenden Punkte sind keine Vermutungen, sondern im Repo nachweisbar:

1. **Gemergte Arbeit ist verloren gegangen.** PR #15 („Simplify project
   structure", −184 Zeilen toter Code, neue `gpu::`-Helfer) ist auf GitHub als
   *merged* markiert — aber auf dem aktuellen `main` **nicht enthalten**
   (`simple_pentagon`, `DrawMesh` etc. existieren noch, die Helfer fehlen).
   Beim manuellen Umschreiben der main-History ist der Merge verschwunden. Die
   Arbeit liegt noch auf dem Branch `claude/wasm-pack-incremental-builds-lN1Mm`.
2. **Merge-Unfälle durch Hin-und-Her:** Der Merge von PR #18 hat eine Zeile aus
   einem früheren Fix zurückgedreht (`Arc` → `Box`) und den WASM-Build
   gebrochen; PR #19 musste das reparieren.
3. **Unaufgeräumte History auf main:** zwei gestrandete `fixup!`-Commits
   (`587fe10`, `8db35ce`), und die Spitze von main ist eine 4-Commit-Debug-
   Kette aus der WASM-Session (inkl. `debug(wasm): log GPU error …`).
4. **Zombie-Branches = vergessene Fäden:**
   - `claude/evaluate-testing-strategy-zxa84`: fertige Unit-Tests für
     Orbital-Mathematik, Sim-Clock, Kamera, Shader-Validierung — nie gemergt.
     Konsequenz: **main hat 0 Tests**, obwohl die Arbeit existiert.
   - `claude/wasm-pack-incremental-builds-lN1Mm`: trägt die verlorene
     #15-Vereinfachung.
5. **PR-Prozess ist Zeremonie ohne Nutzen:** Die meisten PRs wurden Sekunden
   nach dem Öffnen selbst gemergt; zwei PRs (#17, #18) mit Auto-Titeln wie
   „Claude/astronomore wasm render error j2w te" gingen so auf main. Der
   PR-Layer kostet Buchhaltung, liefert aber kein Review.
6. **Ablagerungen:** `claude-code-review.md` im Repo-Root ist ein eingefügtes
   Terminal-Transkript; die Findings sind teils erledigt (sim_time ist längst
   f64), teils noch offen — als Datei im Root aber weder Issue noch Plan.
7. **CI prüft nur main:** Der Pages-Workflow baut erst nach dem Push auf main.
   Der #18/#19-Bruch (reiner WASM-Compile-Fehler) wäre von einem
   `cargo check --target wasm32-unknown-unknown` auf Branches abgefangen worden.

## 4. Diagnose

Das Teure ist nicht das Schreiben von Code — das skaliert mit Claude gut. Teuer
ist, dass **jede Cloud-Session einen neuen Branch, eine neue History und einen
neuen Gesprächsfaden erzeugt**, und die Abstimmung dieser drei Ebenen
(GitHub-Branches, Chat-Threads, CLAUDE.md-Plan) bisher manuell in Antons Kopf
stattfindet. Bei wenig Zeit reißt genau diese Abstimmung zuerst — Resultat:
verlorene Merges, Zombie-Branches, fixup-Reste.

Die Lösung ist nicht mehr Disziplin, sondern: **die Aufräumarbeit dorthin
verlagern, wo sie automatisch passiert** (Claude am Session-Ende, CI), und
**eine einzige Quelle der Wahrheit** für offene Fäden etablieren.

## 5. Empfehlungen — Workflow

### Einmalige Aufräumaktion (~30–45 min)

1. Tests-Branch `claude/evaluate-testing-strategy-zxa84` reviewen und mergen
   (→ main hat endlich Tests).
2. Die #15-Vereinfachung vom Branch `claude/wasm-pack-incremental-builds-lN1Mm`
   auf aktuellem main re-applien (Session-Auftrag an Claude: „rebase & merge,
   Konflikte gegen aktuellen Stand auflösen").
3. `claude-code-review.md` löschen; noch gültige Findings (z. B. Maus-Delta-
   Akkumulation, WASM-Wireframe-Feedback) als GitHub-Issues anlegen.
4. Die `fixup!`-Commits auf main **stehen lassen** — main nicht mehr
   umschreiben; der Schaden ist kosmetisch, das Umschreiben war das Risiko.

### Dauerhafte Regeln (in CLAUDE.md verankert, siehe dort)

- **Claude räumt selbst auf, vor dem Push:** Debug-Hin-und-Her wird per Rebase
  zu logischen Commits gesquasht. Kein `fixup!`, kein `debug:`-Commit, kein
  Auto-PR-Titel verlässt je eine Session. → Antons Squash-Arbeit entfällt.
- **main ist append-only.** History-Korrekturen nur auf Branches vor dem Merge.
  Merge per Fast-Forward oder Squash-Merge-Button — danach nie wieder anfassen.
- **Branch-Lebenszyklus:** Jede Session endet mit gemergt / „bereit zum
  Review" im Journal / gelöscht. Keine stillen Zombies.
- **`docs/JOURNAL.md` als Gedächtnis:** Tabelle offener Fäden, von Claude am
  Session-Ende gepflegt, am Session-Start gelesen. Ersetzt das Erinnern über
  Chat-Threads hinweg. Ein Blick vor jeder Session genügt, um den Stand zu haben.
- **PR nur, wenn er etwas leistet** (CI-Lauf, späteres Nachlesen). Für
  Solo-Merges reicht Branch + ff-Merge; spart die Zeremonie.
- **CI auf Branches:** ein leichter Workflow mit `cargo check` (nativ +
  wasm32) und `cargo clippy -D warnings` für alle Pushes — fängt die
  #18/#19-Klasse von Fehlern, bevor sie main erreicht.

### Zeitbudget-Hinweis

Bei wenig Zeit: **maximal 1–2 offene Fäden gleichzeitig.** Lieber eine
Cloud-Session pro Faden bis „mergebar oder verworfen" treiben, als drei
parallel anstoßen — die Parallelität ist es, die die Buchführungskosten erzeugt.

## 6. Empfehlungen — Lernziele & Pädagogik

Damit Ziel 1 (Computergrafik lernen) nicht unter Ziel 2 (cooles Projekt)
leidet, gilt ab jetzt eine explizite Themen-Grenze (in CLAUDE.md verankert):

- **Antons Territorium (Grafik-Kern):** Shader-/Beleuchtungslogik,
  Transformationen & Projektion, Tessellierung, kommende Render-Techniken
  (Normal-Mapping, Bloom, Atmosphäre, Sternen-Punktwolke).
- **Claudes Territorium:** Boilerplate, egui/QoL, Debugging, WASM/Plattform,
  Tests, Build/CI, Doku.
- **Im Grafik-Kern liefert Claude keine fertigen Implementierungen**, sondern:
  benennt die falsche Annahme, erklärt das Konzept (mit Buchkapitel-Referenz),
  stellt eine Leitfrage — und reviewt Antons Versuch danach konkret.

Für die anstehenden Phase-4/5-Themen heißt das z. B.: Claude baut gern das
HYG-CSV-Parsing und die Punktwolken-Pipeline-Boilerplate — aber die Abbildung
B–V-Index → RGB und die Größenattenuation überlegt sich Anton.
